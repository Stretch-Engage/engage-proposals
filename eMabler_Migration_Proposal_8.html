01 — Scope

What's in and out

IN SCOPE 01

API v2 Migration — All Touchpoints

Trigger → Handler → AsyncProcessor → APIService chain for Charger, Driver, Site, Tag, Socket. Setup Wizard, Permission Manager, Remote Actions, Integration management.

IN SCOPE 02

Trigger Bulkification Refactor

Current architecture processes records individually. Refactor to bulkified upsert pattern — a Salesforce governor limit risk and AppExchange review requirement.

IN SCOPE 03

ECA Transition Planning

Scoping and experimentation for the Salesforce External Client Apps migration. Full package migration targeted as a separate 2027 release — no client impact in this scope.

IN SCOPE 04

AppExchange Re-certification

Security Scanner re-run after auth changes. Regression testing across test and prod environments to meet AppExchange certification standards.

OUT OF SCOPE

Charger Consumptions / Events

API exposed to eMabler for inbound push. No v2 changes required — confirmed out of scope in the proposal.

OUT OF SCOPE

Full ECA Package Migration

Converting ConnectedApp metadata to ExternalClientApp format is a 2027 major release item. Only planning and experimentation in this scope.

OUT OF SCOPE

Client Org Data Migrations

Any groupId → siteId field value migrations in existing client orgs are out of scope and must be handled per client.

03 — Breaking Changes

What needs rebuilding

🔐 Authentication — all callouts

V1 used a dual-key scheme (apiKey + customerKey). V2 requires a single x-api-key header. Old keys are not valid for v2 — new keys must be generated from eMabler Connect. The Named Credential and/or Custom Metadata storing auth config must be rebuilt from scratch.

🏷️ group → site rename — all objects

The concept of a "group" is now a "site" everywhere. Every Apex DTO class mapping groupId, groupName, homeGroupId, parentGroupId must be updated. Any Salesforce field mappings storing these values need a data migration plan for existing client orgs.

⚡ Remote actions — GET → POST

All five remote actions (Start, Stop, Reboot, Unlock Cable, Refresh Config) have changed from HTTP GET with parameters in the URL path to HTTP POST with a structured JSON body. Each callout's request construction must be rebuilt. Responses also change from plain text to structured JSON.

📄 Response codes throughout

V2 standardises on 201 Created (not 200) for all create operations, and 204 No Content (not 200 with plain text) for deletes. Response parsing logic must handle these codes and must not expect a response body on delete operations.

📑 Pagination on all list endpoints

V1 returned plain arrays. V2 wraps all list responses in a paginated envelope { page, limit, total, items: [...] }. Every list callout must be updated to loop through pages until page × limit ≥ total. This is a structural change to the sync architecture, not just a field rename.

👤 Driver path & field changes

Driver endpoints move from /authorise/user/ to /v2/drivers/. The lookup key changes from externalId to driverId. Field homeGroup becomes homeSiteId. integratedExternalIds becomes externalUserIds. Responses expand from plain text / id-only to full structured objects.

Open questions for eMabler

These must be resolved before development begins:

  • Bulk driver query: Does GET /v2/drivers support querying multiple drivers in a single request? In Salesforce bulkified contexts, the connector may have 200 driver IDs to resolve simultaneously — individual round-trips would be prohibitive.
  • Max chargers/sites per driver: What is the maximum number of chargers or sites assignable to a single driver? Determines whether the Permission Manager needs its own bulk handling strategy.
  • Firmware & Diagnostics: Present in v1 (POST /charger/firmware/{id}, POST /charger/diagnostics/{id}) but not found in the v2 Entity Management spec. eMabler to confirm whether these moved to a v2 sub-package (e.g. Express API) or are handled differently in v2.
  • Payment Terminals & Operators: Full CRUD present in v1 but not found in the v2 Entity Management spec. eMabler to confirm whether these moved to another v2 sub-package, or whether the connector is expected to use them at all.

06 — Salesforce Platform

External Client Apps transition

Salesforce is retiring the ability to manually create legacy Connected Apps in the Setup UI from Spring '26. AppExchange Managed Packages are explicitly exempt — the connector continues to work for new and existing client installs. However, the team's dev workflow and long-term architecture need to adapt.

Legacy vs External Client App

FeatureLegacy Connected AppExternal Client App (ECA)
PackagingSingle XML fileModular: App + OAuth + Policy files
DistributionTied to publishing orgGlobal OAuth Settings — better portability
SecurityScopes defined at creationGranular policies; admin enables post-install
UI creationRetired Spring '26 (manual UI only)Supported going forward
Managed package installsExempt — safe for new installsTarget format for next major version
CI/CD deploymentsf project deploy start still worksSupported via CLI

Recommended roadmap

1
Now → Winter '26

Secure the baseline

Ensure all ConnectedApp configurations are in source control. Stop any manual Connected App creation in dev orgs. All dev environment setup must go through metadata deployment.

2
Mid 2026

Experiment with ECA

Use Salesforce CLI to convert existing Connected App metadata to ECA format in scratch orgs. Validate the callback flow. No client impact — experimental only.

3
2027 roadmap

Release ECA package version

Ship a package update including the ECA version alongside the legacy Connected App. Salesforce supports both during the transition — no downtime for existing client charger callbacks.

Reference: Salesforce Knowledge Article 005228017 and ISV partner guidelines on the 2026 retirement of manual legacy app creation.

07 — Key Success Criteria

How we work together

A successful migration is as much about collaboration as it is about code. These are the working agreements and structural requirements we need in place from day one — for both the project and the ongoing LCM.

📋 Shared Jira — single source of truth

All work — project tickets, LCM tasks, bugs, backlog items — lives in one shared Jira project that both teams have access to.

  • Project board tracks the migration sprint by sprint with clear acceptance criteria per ticket
  • LCM board tracks ongoing maintenance, open questions, and discovered issues with full visibility
  • Backlog captures the roadmap items from this proposal (Agentforce, Data Cloud, ECA) as groomed future epics
  • No work happens outside Jira — if it's not a ticket, it doesn't exist

💬 Slack — joint team channel

A dedicated shared Slack channel between Stretch Engage and eMabler as the primary support and communication channel for the connector.

  • Day-to-day questions — faster than email, creates a searchable record
  • Incident handling — production issues raised and tracked in Slack before a Jira ticket is created
  • Jira and GitHub notifications piped into the channel so both teams see progress in real time
  • LCM support — the channel is the support entry point for eMabler's team during the retainer

🔍 Open questions resolved before dev starts

Several technical questions to eMabler remain open and block accurate scoping. These must be answered before the project kicks off.

  • Does GET /v2/drivers support bulk queries? (Salesforce bulkified context requires this)
  • Max chargers/sites per driver — determines Permission Manager bulk strategy
  • Firmware, Diagnostics, Payment Terminal endpoints — in v1 but not found in v2 Entity Management spec; confirm location in v2 with eMabler
  • Access to a shared test environment with realistic data for regression testing

✅ Definition of done — migration

The project is complete when all of the following are true — not before.

  • All active connector endpoints migrated and verified against v2 in test environment
  • Bulkification refactor merged and governor limit tests passing
  • AppExchange Security Scanner re-run with no new critical findings
  • Package version submitted and approved for AppExchange listing update
  • Regression test suite passing in both test and production environments
  • eMabler team sign-off on end-to-end flow validation
Transparency over updates

We don't wait for status meetings to share progress. Jira is live, Slack is open, and both teams can see what's in progress, what's blocked, and what's done at any time. No surprises.

One backlog, one team

There is no "Stretch backlog" and "eMabler backlog". There is one shared product backlog for the connector. Both teams contribute, both teams groom it, both teams own the outcome. Shared accountability.

LCM is proactive, not reactive

The monthly retainer is not a break-fix service. Each month's hours are planned in advance — maintenance, improvements, and backlog progress. No wasted capacity.

08 — Recommended Approach

How to execute the migration

  1. Code audit (1 day) — Walk the connector codebase and confirm which API endpoints are actively called. This de-risks the estimate, removes ambiguity, and is worth doing as a paid first step before committing to a fixed scope.
  2. Resolve open questions with eMabler — Particularly bulk driver querying and the location of Firmware, Diagnostics, and Payment Terminal endpoints which existed in v1 but are absent from the v2 Entity Management spec. These affect scope.
  3. Auth and base URL migration first — Unblocks all subsequent testing against v2 endpoints. Must be done before anything else can be validated.
  4. Domain-by-domain migration — Site (including the group→site rename), Charger, Driver, Tag, Integration, Remote Actions. Each domain is independently testable and deployable.
  5. Bulkification refactor — Tackle during or immediately after domain migration while the code is open. Deferring to a separate sprint increases cost and risk.
  6. Regression testing and AppExchange security re-scan — Full regression across test and production environments before release submission. Allow time for re-review if auth changes trigger it.

Custom conversational UI

Not the right investment at this stage. Agentforce handles the conversation layer — the connector's role is to expose reliable actions that agents can invoke, not to build a UI on top.

Real-time socket-level telemetry streaming

Deferred until the Data Cloud foundation (Phase 2) is in place. Building this before the data model is defined creates rework and tech debt.

Loyalty Cloud

Deferred until Phase 2 (Data Cloud) is complete and there is validated customer demand. Loyalty Cloud without a data foundation has no analytics backbone to build on.

Feature expansion before v2 is complete

Any new functionality built on v1 increases migration complexity. The v2 migration is the prerequisite for everything else — new features come after the foundation is solid.
