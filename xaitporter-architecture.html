Executive Summary

The XaitPorter Connector is a Salesforce managed package (namespace: xaitporter) that bridges Salesforce CRM data with the XaitPorter document-collaboration and tender-management platform. It allows sales teams to initiate, link, and synchronise XaitPorter projects directly from Salesforce Account and Opportunity records, eliminating manual data re-entry and keeping both systems in sync.

Functional Overview

What the connector does

When a salesperson is working on a Salesforce Account or Opportunity, they can use the connector to:

  1. Browse available XaitPorter project templates relevant to that record.
  2. Duplicate a template to create a new XaitPorter project, pre-populated with Salesforce data.
  3. Have the new project linked back to the Salesforce record as a child XaitporterProject__c object.
  4. Update project metadata in XaitPorter when Salesforce record data changes (manually or via Flow).
  5. Open the project URL directly in XaitPorter from the Salesforce UI.

Administrators configure which Salesforce fields map to which XaitPorter metadata fields, what format dates, booleans, and multiselect values use, and which XaitPorter project roles map to which Salesforce roles.

Authentication & Security

Authentication model

The connector uses per-user OAuth 2.0 (OpenID Connect) via Salesforce Named Credentials and External Credentials. Every Salesforce user who interacts with XaitPorter must independently authorise the connection — there is no shared service account for API calls.

Authentication components

ComponentTypeRole
XaitPorter_REST_API Named Credential Holds the API base URL and per-user OAuth tokens. Protocol: OAuth, Principal: PerUser. Referenced by XPAPI_Client for all REST callouts.
Auth_API_Ext External Credential Manages per-user OAuth principals. Linked to xaitPorterTest1 Auth Provider. Used for ExternalDataUserAuth checks in setup flow.
xaitAuthProvider Auth Provider (OpenID Connect) Primary auth provider. Authorization and token endpoints at salesforce-test.xaitporter.com. Consumer Key: "Spreadsheet Tool". Sends access token in header. PKCE: disabled. MFA: not required.
xaitPorterTest / xaitPorterTest1 Auth Provider Additional auth provider configurations for test environments.

Per-user authentication flow

1. User opens XaitPorter component on a record page
↓
2. XaitSettingsCmpController.getSettingData() checks ExternalDataUserAuth for this user
↓
3. If not authenticated → initiateBrowserAuthFlow() redirects to xaitAuthProvider authorize URL
↓
4. User logs in to XaitPorter → OAuth token returned and stored against the Named Credential (per-user principal)
↓
5. All subsequent API calls use the Named Credential — Salesforce automatically injects the Bearer token

Apex Classes

Controller
XaitCreateProjectWithTemplateController global with sharing
Core project creation controller. Fetches templates, loads field mappings, calls the XaitPorter API to duplicate a project, and writes the resulting XaitporterProject__c record. Exposes an @InvocableMethod (updateProjectInvocable) for Flow integration. Handles date/datetime/boolean/multiselect formatting per the configured custom metadata.
Controller
XaitSettingsCmpController
Admin settings controller. Tests API connectivity (/me), fetches and syncs project roles (/projectroles), and deploys XaitPorter_Setting__mdt and XaitPorter_ProjectRole_Mapping__mdt records via Metadata.DeployContainer. Enforces XaitPorter_Admin_Permissions permission set.
Controller
XaitTemplatesCmpController
Template management controller. Retrieves templates from XaitPorter API, persists them as XaitporterTemplate__c, manages field mappings (SFDC_Xaitporter_Field_Mapping__c) and template filters (XaitporterTemplateFilter__c). Performs object introspection to present available Salesforce fields to the admin.
Controller
XaitAppSetupController
User onboarding and permission assignment controller. Lists active licensed users, assigns/removes XaitPorter_User_Permissions permission sets, checks authentication state via ExternalDataUserAuth, and initiates OAuth browser auth flow via ConnectApi.NamedCredentials. Requires ModifyAllData for permission assignment.
Handler
XaitPorterProjectHandler
Trigger handler. Enforces the business rule that only one XaitporterProject__c record per parent Account or Opportunity can have Primary__c = true.
API Service
XPAPI_Service
Business-level API client. One method per XaitPorter endpoint: meGet(), projectrolesGet(), projectsGet(), projectsPost(), projectsUpdateMetadata(), usersGet().
API Client
XPAPI_Client
HTTP transport layer. Configures Named Credential callout, sets headers and 120s timeout. Parses structured XPExceptionResponse errors from the XaitPorter API.
Utility
XPAPI_Utility
Shared API utilities. Defines Param, Authentication interface, ApiKeyHeaderAuth, ApiKeyQueryAuth, and ApiException.
Utility
Safely
Safe DML wrapper. Chainable methods for insert/update/delete/upsert with field-level and object-level permission checks. Used throughout to avoid DML exceptions.
Test Classes
CanTheUserTest · CreateProjectWithTemplateCtrlTest · SafelyTest · TestFactoryAccess
Unit and integration test classes covering permission checks, project creation, and DML utilities.

Triggers

Trigger
XaitPorterProjectTrigger after insert · after update
Fires on XaitporterProject__c. Delegates to XaitPorterProjectHandler.handlePrimaryFlag() to enforce the one-primary-project-per-parent rule. Follows the trigger handler pattern — no business logic in the trigger itself.

Project Creation Flow

End-to-end walkthrough of a user creating a XaitPorter project from an Opportunity record:

1. Entry point — User clicks "Create XaitPorter Project" Quick Action on Opportunity record
↓
2. Auth check — Component verifies user has authenticated via Named Credential (ExternalDataUserAuth). If not → OAuth browser flow.
↓
3. Template fetch — fetchTemplateData() loads XaitporterTemplate__c records filtered for Opportunity context, applying any XaitporterTemplateFilter__c rules.
↓
4. User selects template — UI presents filtered templates. User selects one.
↓
5. Load metadata — loadTemplateMetadata() reads SFDC_Xaitporter_Field_Mapping__c records, queries Salesforce source fields (Account/Opportunity/Contact/Owner), applies format transforms.
↓
6. API call — POST /projects — duplicateProject() → XPAPI_Service.projectsPost() → HTTP POST via Named Credential. Body includes template ID and project metadata.
↓
7. API call — PATCH /projects/:id/metadata — updateProject() pushes field-mapped Salesforce values to the new project's metadata fields.
↓
8. Salesforce record created — XaitporterProject__c inserted with Project_ID__c, Project_URL__c, parent lookup, and Primary__c.
↓
9. Trigger fires — XaitPorterProjectTrigger → XaitPorterProjectHandler.handlePrimaryFlag() ensures only one project is marked primary on this Opportunity.

Authentication Setup Flow (Admin)

1. Admin opens XaitPorter Settings component (requires XaitPorter_Admin_Permissions)
↓
2. getSettingData() checks Auth_API_Ext external credential status and loads XaitPorter_Setting__mdt
↓
3. Admin clicks "Test Connection" → performGetMe() calls GET /me via Named Credential
↓
4. Admin clicks "Sync Roles" → performGetProjectRoles() calls GET /projectroles, maps to XaitPorter_ProjectRole_Mapping__mdt
↓
5. Admin saves format settings (date/boolean/multiselect) → saveSettings() deploys XaitPorter_Setting__mdt via Metadata API
↓
6. Admin assigns users → XaitAppSetupController.saveChanges() assigns/removes XaitPorter_User_Permissions permission set (requires ModifyAllData)
↓
7. Each user follows OAuth browser flow on first use → token stored per-user in Named Credential

Security Posture

Sharing model

  • XaitCreateProjectWithTemplateController is declared global with sharing — respects the running user's sharing rules.
  • SOQL queries mix USER_MODE (for field-level security enforcement) and SYSTEM_MODE where elevated access is legitimately required.
  • Safely utility class enforces object and field CRUD/FLS before all DML operations.

Outbound network security

  • All external HTTP calls go through a Named Credential — no credentials ever appear in Apex code.
  • Named Credential uses per-user OAuth 2.0 — no shared service account tokens.
  • Remote Site Setting ApexDevNet restricts outbound destinations.
  • Access token is injected by Salesforce automatically in the Authorization header — developers cannot read it.

Open items for Security Review

⚠️ Test Auth Provider endpoints xaitAuthProvider, xaitPorterTest, and xaitPorterTest1 point to salesforce-test.xaitporter.com. Verify production endpoints are configured before go-live.
⚠️ Named Credential base URL XaitPorter_REST_API currently points to salesforce.lab.xait.no (lab environment). Must be updated for each target environment.
✅ No hardcoded credentials All authentication is handled through Named Credentials and External Credentials. No secrets in Apex code.
✅ Permission-gated admin operations Admin functions require XaitPorter_Admin_Permissions permission set and/or ModifyAllData.

Permissions Model

Permission SetTarget UsersGrants
XaitPorter_User_Permissions All XaitPorter connector users Access to custom objects, components, Quick Actions. Required for Named Credential per-user auth.
XaitPorter_Admin_Permissions XaitPorter configuration admins Access to Settings component, role sync, metadata deployment. Required for XaitSettingsCmpController.

Assignment of XaitPorter_User_Permissions is managed via the XaitPorter App Setup component, which requires the assigning admin to have ModifyAllData. Bulk assignment to multiple users is supported.
